# Attrax 三人 24 小时执行计划

## 分工原则

比赛只剩 24 小时，三人版分工按“闭环优先”执行：

- Person A：前端工程主力，负责 `frontend/screen-display` 页面、3D 状态映射、交互联调。
- Person B：后端和智能逻辑主力，负责 `/api/attrax/home/*`、规则解析、SSE、reset 和演示稳定性。
- Person C：视觉和展示主力，负责 UI 观感、贴图优先级、PPT、讲稿、录屏截图和答辩叙事。

## 当前已落地内容

- 后端新增 `backend/attrax_home.py`。
- 主服务 `backend/main.py` 已挂载 `/api/attrax/home/*` 和 `/attrax-display` 静态前端入口。
- 前端已改为直接增强 `frontend/screen-display/house-detail.html` 和 `frontend/screen-display/house-detail.js`，不再保留独立 mock 页面。
- 文档蓝图已写入 `Attrax_competition/pro_blueprint.md`。

## 0-4 小时：完成可运行骨架

Person A：

- 完成三栏页面：聊天区、3D 展示区、设备和建议区。
- 接入初始状态读取 `GET /api/attrax/home/state`。
- 加入四个快速演示按钮：空调 23 度、灯光 30%、窗帘一半、观影模式。

Person B：

- 完成 mock 设备状态：空调、灯光、窗帘、环境传感器。
- 完成设备命令接口和聊天接口。
- 完成 SSE 事件流和 reset 接口。

Person C：

- 明确展示主题：AI 让家庭空间“有回应”。
- 设计 PPT 结构：痛点、方案、架构、Demo、商业/落地价值。
- 确定视觉基调：深色展示屏、蓝色科技感、暖色家庭反馈。

交付物：

- 后端能启动。
- 前端能打开。
- 四个按钮至少能更新设备卡片。

## 4-8 小时：打通聊天到设备执行

Person A：

- 聊天表单接入 `POST /api/attrax/home/chat`。
- 设备卡片加入滑块控制：温度、灯光亮度、窗帘开合比例。
- 显示后端 reply、当前场景、连接状态。

Person B：

- 完成中文规则解析：
  - “把空调调到 23 度”
  - “灯光调到 30%”
  - “窗帘打开一半”
  - “进入观影模式”
- 场景联动策略：观影、睡眠、离家、欢迎。
- 每次执行后生成 `device.command_applied` 事件。

Person C：

- 完成第一版 PPT。
- 准备产品命名、口号和 3 个用户故事。
- 给前端补充设备文案和建议文案。

交付物：

- 聊天输入能驱动设备状态变化。
- 后端事件时间线能展示解析和执行过程。

## 8-12 小时：强化 3D 可视化和智能建议

Person A：

- 把灯光亮度映射到室内点光源和发光效果。
- 把空调状态映射到 3D 空调和冷气流。
- 把窗帘比例映射到窗帘开合动画。
- 房屋屏幕显示温度、光照、设备摘要。

Person B：

- 完成智能建议规则：
  - 室温偏高 -> 建议空调 23 度。
  - 光照不足 -> 建议灯光 65%。
  - 阳光过强 -> 建议窗帘 45%。
- 补充 API health 和 reset 稳定性。
- 保证所有命令失败时返回清晰错误。

Person C：

- 制作架构图：前端展示层、后端中枢、设备 adapter、真实硬件。
- 制作交互流程图：用户意图 -> 规则/LLM -> 设备动作 -> 实时反馈。
- 开始录制阶段性演示素材。

交付物：

- 3D 场景和设备状态同步。
- 智能建议可点击执行。

## 12-16 小时：完整联调

Person A：

- 联调 SSE，确保实时事件能刷新 UI。
- 加 SSE 失败后的 2 秒轮询降级。
- 修复不同屏幕比例下的布局问题。

Person B：

- 全链路联调：聊天 -> 解析 -> 命令 -> 状态 -> SSE -> 前端。
- 固定 demo 初始状态，保证 reset 后每次演示一致。
- 准备命令行或接口测试样例。

Person C：

- 把最新页面截图放入 PPT。
- 完成 3 分钟讲稿初稿。
- 准备评委问题回答：为什么是赛道二、如何接硬件、AI 价值在哪里。

交付物：

- 四个核心 demo case 连续跑通。
- PPT 有完整故事线。

## 16-20 小时：稳定性和展示打磨

Person A：

- 修复前端黑屏、模型加载失败、按钮重复触发、文本溢出等问题。
- 加载 GLB 失败时保证简化 3D 房屋兜底可见。
- 优化展示屏上的状态文字。

Person B：

- 做接口异常测试。
- 确认 reset、state、chat、command、events 都可用。
- 准备真实硬件 adapter 的口头说明，不在最后阶段强行接入高风险设备。

Person C：

- 完成最终 PPT。
- 完成现场演示分镜。
- 准备备用截图和短视频，防止现场网络或设备异常。

交付物：

- 一套稳定现场演示。
- 一套备用离线讲解材料。

## 20-24 小时：只修阻塞问题

Person A：

- 不加新功能，只修影响演示的前端 bug。
- 配合 Person C 做最终截图/录屏。

Person B：

- 不改接口结构，只修启动和状态问题。
- 保证 `python backend/main.py` 可以启动后端。

Person C：

- 组织至少 3 次完整彩排。
- 控制讲解节奏，确保 3 分钟版本和 1 分钟版本都能讲完。
- 整理提交材料。

交付物：

- 最终可演示页面。
- 最终 PPT。
- 备用截图/录屏。
- 明确每个人现场负责的话术。

## Demo 验收清单

- 输入“把空调调到 23 度”，空调卡片、3D 冷气流、屏幕温度状态同步变化。
- 输入“灯光调到 30%”，灯光卡片、3D 发光强度同步变化。
- 输入“窗帘打开一半”，窗帘卡片、3D 窗帘位置同步变化。
- 输入“进入观影模式”，灯光、窗帘、空调联动变化。
- 点击 reset 后能重新从固定状态开始演示。
- 后端断开时前端不崩溃，能显示兜底状态或连接提示。
