# homekgmas 项目介绍

`homekgmas` 是一个面向智能家居场景的本地优先多智能体研究原型。它不是单纯的概念性脚手架，而是一套已经可以跑通端到端流程的系统：接收任务、读取家庭状态、唤醒相关领域代理、开展讨论、解决冲突、生成执行计划、驱动模拟器执行，并将结果以本地文件形式保存下来。

这个仓库同时承担两类角色：

- 作为一个可运行的智能家居编排系统，用于验证多代理协作控制流程。
- 作为一个研究平台，用于数据集构建、离线评测、记忆机制对比和运行基准测试。

## 项目定位

项目围绕“智能家居管家”这一目标展开，重点解决以下问题：

1. 如何把用户请求、定时任务或上下文触发的任务转成结构化协同问题。
2. 如何从家庭环境状态中识别出应被唤醒的领域代理。
3. 如何让多个代理提出方案、发现冲突并收敛为一致行动。
4. 如何在不依赖数据库和云端基础设施的前提下完成本地记忆持久化。
5. 如何为后续数据集构建、评测和记忆实验保留可复现的运行痕迹。

## 当前已经具备的能力

当前代码已经实现完整主链路，默认运行模式为 `fusion`，默认环境为嵌入式模拟器：

1. 接收任务请求或调度任务。
2. 读取室内传感器、室外环境和设备状态。
3. 构建讨论主题与上下文。
4. 选择最相关的领域代理参与。
5. 为参与代理检索本地记忆。
6. 生成多代理提案。
7. 检测策略冲突或动作冲突，并进行一轮修订。
8. 汇总为协调后的执行计划。
9. 在模拟器中执行动作。
10. 将任务结果、工作区笔记和三元组记忆写入磁盘。

## 支持的代理领域

项目最初围绕三个核心代理展开，但当前运行时已经扩展为以下八类领域代理：

- `cooling_agent`
- `lighting_agent`
- `music_agent`
- `fan_agent`
- `cover_agent`
- `lock_agent`
- `switch_agent`
- `appliance_agent`

这些代理通过注册表统一装配，具备各自的能力边界、工作区和提案逻辑。在启用 LLM 时，代理可结合模型生成更丰富的提案；在未启用 LLM 时，系统仍可基于本地规则运行。

## 核心架构

项目的主运行入口位于 `app/main.py`。应用启动时会完成以下装配：

- 加载系统配置与输出目录。
- 初始化 LLM 客户端（可选）。
- 初始化三元组记忆与工作区记忆。
- 初始化智能家居模拟器。
- 根据代理模式加载代理目录与注册表。
- 创建 `CentralNode` 作为统一编排中枢。
- 暴露 FastAPI 接口供任务提交、上下文查看和调度使用。

主运行链路如下：

`Task Request -> CentralNode -> WakeupManager -> DiscussionManager -> Consensus/Resolution -> ExecutionPlan -> Simulator -> Memory Persistence`

其中关键模块包括：

- `app/orchestrator/central_node.py`：总控节点，负责把感知、讨论、规划、执行和记忆串起来。
- `app/orchestrator/wakeup_manager.py`：根据任务和上下文计算代理唤醒分数。
- `app/orchestrator/discussion_manager.py`：组织代理提案与修订流程。
- `app/discussion/conflict_detector.py`：检测动作或策略冲突。
- `app/discussion/action_resolver.py`：将多个提案整合为协调后的动作集合。
- `app/environment/`：提供嵌入式和远程两种模拟器后端。
- `app/memory/`：提供图式记忆、工作区记忆、检索与压缩能力。

## 本地优先设计

这个项目刻意保持轻量、可检查和可复现：

- 默认使用文件系统持久化，不依赖数据库。
- 不引入 Redis、Neo4j、PostgreSQL、Celery、Kubernetes 或微服务体系。
- 运行结果尽量落到 `outputs/` 下，便于调试、回放和评测。
- 记忆体系同时支持三元组记录、工作区笔记和对话压缩。
- 模拟器既可以内嵌在主服务里，也可以独立启动为本地 HTTP 服务。

这意味着它更适合作为本地研发和研究实验平台，而不是直接面向生产环境的大规模基础设施。

## 记忆与输出

当前项目的记忆系统由多个本地组件协同完成：

- `TripleStore`：保存任务摘要和事实型记录。
- `WorkspaceMemoryStore`：保存代理短期/长期工作区笔记。
- `GraphRetriever`：从历史记录中抽取与当前任务相关的图式上下文。
- `DialogueCompressor`：压缩讨论历史和未解决冲突，减少上下文负担。

常见输出包括：

- `outputs/memory/records/*.json`
- `outputs/memory/triples.jsonl`
- `outputs/agent_workspaces/fusion/*/memory/*.jsonl`
- `outputs/logs/`
- `outputs/reports/`

## 数据与评测能力

除了运行时编排系统，仓库还保留了较完整的研究面：

- 多来源智能家居数据集的下载、清洗与统一建仓。
- 统一任务样本和 episode 结构的构建。
- 离线评测、基准测试与 memory backend 对比。
- API 驱动的数据采集与结果分析。

相关能力主要分布在以下目录：

- `app/datahub/`：数据仓库结构、校验和质量控制。
- `app/datasets/`：统一数据适配与任务构建。
- `app/evaluation/`：基准测试、数据集评测和融合基线。
- `scripts/`：构建数据、运行评测、启动服务和生成报告。

因此，这个仓库既能用于“跑系统”，也能用于“做实验”。

## 目录概览

项目中最值得优先理解的目录如下：

- `app/`：核心应用代码。
- `configs/`：系统、设备、传感器与代理配置。
- `scripts/`：运行服务、模拟器、评测和数据构建脚本。
- `tests/`：核心模块测试。
- `reports/`：数据画像、评测摘要和分析文档。
- `screen-display/`：可视化展示相关静态资源。

## 快速运行

推荐使用 Python 3.10+。

```bash
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
python scripts/run_server.py
```

启动后可访问：

- `http://127.0.0.1:8000/`
- `http://127.0.0.1:8000/health`
- `http://127.0.0.1:8000/docs`

常用接口包括：

- `POST /api/v1/tasks/demo`
- `GET /api/v1/tasks/context/current`
- `POST /api/v1/tasks/scheduled`
- `POST /api/v1/tasks/scheduled/run-due`

如果需要单独运行模拟器服务，可执行：

```bash
python scripts/run_simulator_server.py
```

## 当前边界

虽然系统已经可运行，但它仍然保持研究原型的特征：

- 执行规划器目前较轻量，主要负责把达成一致的动作向下游传递。
- 讨论流程目前以初始提案加一轮修订为主，还不是复杂的多轮协商系统。
- 检索和记忆选择仍以本地启发式策略为主，没有引入重型语义检索基础设施。
- 调度器当前为内存实现，适合原型验证，不是持久化任务系统。

## 一句话总结

`homekgmas` 是一个已经具备端到端能力的本地优先智能家居多智能体原型：它把模拟环境、多代理协作、本地记忆、数据集构建和评测研究放在同一个仓库里，重点验证“可运行、可观察、可评测”的智能家居协同控制流程。

## 可视化展示现状与待补齐项

仓库中的 `screen-display/` 已经包含较完整的三维展示原型，包括城市总览、房屋细节、能源追踪和爆炸图页面，也包含多份 `.glb` 模型、天空盒资源以及基于 Three.js 的复杂场景脚本。从视觉资产和交互表现来看，这一部分已经具备“展示原型”的基础，但从当前项目整体实现来看，它还没有完全接入主运行时，距离“可运行的系统可视化前端”还差几层关键能力。

### 当前已经具备的部分

`screen-display/` 目前已经具备：

- 基于 Three.js 的三维场景与镜头控制。
- 城市级和房屋级两层可视化页面。
- 房屋部件拆解、能源流动、昼夜切换等高表现力展示效果。
- 本地模型、纹理、天空盒与场景交互脚本。
- 一部分用于调节相机、材质、对象位置的开发者调参界面。

换句话说，当前缺的不是画面本身，而是把这些画面接入现有智能家居运行时的数据链路、交互链路和产品链路。

### 还缺少的关键部分

#### 1. 后端状态到三维场景的适配层

当前后端已经能提供家庭上下文和设备状态，例如：

- 传感器温度、湿度、光照、占用情况。
- 空调、灯光、风扇、窗帘、门锁、开关设备和家电状态。
- 任务执行后的动作结果与状态快照。

但 `screen-display/` 目前主要还是本地场景逻辑，并没有形成稳定的数据适配层。要真正做成项目展示前端，还需要把运行时状态映射到场景对象，例如：

- 灯光亮度映射为室内照明或材质发光。
- 窗帘位置映射为模型开合状态。
- 锁状态映射为门锁视觉反馈。
- 音乐、风扇、空调等设备映射为动态提示或局部动画。
- 任务执行结果映射为页面中的设备变化和状态更新。

#### 2. 实时数据更新机制

当前展示页没有形成明确的实时数据通路。若要展示智能家居系统的实时运行过程，还需要至少一种更新机制：

- 定时轮询后端接口。
- WebSocket 推送。
- Server-Sent Events 事件流。

否则页面只能展示静态场景或一次性初始化状态，无法体现“任务发出后，代理讨论、计划执行、设备状态随之变化”的过程。

#### 3. 面向用户的业务界面

现在的展示页包含大量开发期调试能力，例如对象位置调整、材质参数调节、相机导出等。这些对开发很有价值，但还不等于面向最终展示的业务界面。

如果要让它成为真正的项目展示前端，还需要补上面向用户的业务信息层，例如：

- 任务输入框或演示任务按钮。
- 当前家庭状态摘要。
- 当前被唤醒的代理列表。
- 代理讨论与冲突提示。
- 执行动作时间线。
- 最近事件或最近任务结果面板。

这样用户才能理解系统当前在做什么，而不只是看到一个好看的三维房屋模型。

#### 4. 与主服务的页面集成

当前主应用由 FastAPI 驱动，主要暴露任务和上下文接口；而 `screen-display/` 更像一组单独运行的静态页面。要把它真正纳入项目，还需要补一层正式集成方式，例如：

- 通过 FastAPI 挂载静态页面目录。
- 或者给 `screen-display/` 单独提供前端启动方式和明确的 API 地址配置。

如果缺少这一步，它就仍然是“仓库里的独立展示样机”，而不是项目本身的一部分。

#### 5. 可视化叙事与系统主链路的对应关系

当前系统主链路是：任务进入、选择代理、讨论提案、冲突处理、计划执行、记忆落盘。

如果想把展示层做完整，还需要定义清楚不同页面分别承担什么职责，例如：

- 城市页展示总体运行状态。
- 房屋详情页展示设备和空间状态。
- 能源页展示能源相关指标和流向。
- 专门的协同页展示代理唤醒、提案和冲突解决过程。

只有把视觉页面和系统主链路对齐，可视化才会形成完整的产品叙事，而不是若干彼此独立的炫技页面。

#### 6. 能源展示的数据基础

仓库里已经有 `energy-tracking` 页面，但当前主运行时的环境状态并没有形成一套完善、统一的能源流模型。若想让能源展示不是单纯动画效果，而是和系统运行一致，还需要补：

- 统一的能耗或能量状态字段。
- 设备级或房间级的能耗统计逻辑。
- 面向前端的能源接口输出。

否则能源页只能停留在概念展示，而难以成为真实系统状态的一部分。

#### 7. 前端工程化整理

目前展示脚本体量已经很大，特别是 `house-detail.js` 和 `app.js`，继续在单文件里叠加逻辑会越来越难维护。后续如果要继续实现，建议补齐：

- 场景逻辑、数据逻辑、UI 逻辑的拆分。
- 统一的资源加载与错误处理。
- 可复用的状态同步模块。
- 前端工程配置，例如模块化构建、依赖管理和发布方式。

这一步不是最先要做的，但如果不做，展示层会很快变成难以维护的大脚本。

### 建议的实现优先级

如果目标是尽快把当前展示做成一个“能接项目、能演示系统”的版本，建议优先做下面四件事：

1. 先把展示页接入 `GET /api/v1/tasks/context/current`，实现设备与环境状态可视化。
2. 再接入 `POST /api/v1/tasks/demo`，实现从页面直接触发任务。
3. 为页面增加任务结果、已选代理、冲突和执行动作的展示面板。
4. 把 `screen-display/` 正式挂到主服务或独立前端入口上，形成稳定运行方式。

### 一句话判断

当前 `screen-display/` 已经是一个质量不低的三维展示原型，但它目前更像“视觉样机”，还不是“系统前端”。真正还差的，主要不是模型和特效，而是与现有多智能体运行时之间的数据接入、实时同步、业务界面和工程化集成。
